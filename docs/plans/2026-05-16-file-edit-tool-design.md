# file_edit Tool Design

## Problem

The project's only file mutation tool is `file_write`, which does a full overwrite via `os.WriteFile`. When an LLM agent needs to make a targeted change to a file, it must read the entire file, reconstruct the modified version in its response, and write it back. This is expensive in tokens, error-prone for large files, and a common source of hallucinated or dropped content.

## Solution

Add a `file_edit` tool to `internal/tools/` that supports three targeted edit operations in a single tool with a flat JSON schema.

## Schema

```json
{
  "type": "object",
  "properties": {
    "path":       { "type": "string" },
    "operation":  { "type": "string", "enum": ["replace_text", "replace_lines", "apply_patch"] },
    "old_string": { "type": "string" },
    "new_string": { "type": "string" },
    "start_line": { "type": "integer" },
    "end_line":   { "type": "integer" },
    "content":    { "type": "string" },
    "patch":      { "type": "string" }
  },
  "required": ["path", "operation"]
}
```

All operation-specific parameters are optional at the schema level. Missing required params for a given operation are caught at runtime with a clear error message.

## Operations

### replace_text

Required: `old_string`, `new_string`

Reads the file, replaces the first occurrence of `old_string` with `new_string`, writes back. Returns an error if `old_string` is not found. Only the first occurrence is replaced; the LLM can call again for subsequent ones.

### replace_lines

Required: `start_line`, `content`

Optional: `end_line` (defaults to `start_line` if omitted)

Lines are 1-indexed. Replaces lines `start_line` through `end_line` inclusive with `content`. Providing empty `content` deletes the targeted lines. Returns an error if line numbers are out of range or `end_line < start_line`.

### apply_patch

Required: `patch`

Applies a unified diff patch (standard `--- a/`, `+++ b/`, `@@ ... @@` format). Implemented as a pure Go string-based hunk parser — no external process, no cgo. Returns an error if any hunk fails to apply.

## Implementation

**File**: `internal/tools/file_edit.go`

`Execute()` unmarshals all params into a single flat struct, dispatches on `operation` via a `switch`, and calls a private method per operation: `replaceText`, `replaceLines`, `applyPatch`.

**Registration**: `FileEditTool` registered in `internal/tools/registry.go` alongside existing tools.

## Error Handling

| Operation | Error condition | Message |
|---|---|---|
| `replace_text` | `old_string` absent | `old_string not found in <path>` |
| `replace_lines` | `start_line` out of range | `start_line N out of range (file has M lines)` |
| `replace_lines` | `end_line < start_line` | `end_line must be >= start_line` |
| `apply_patch` | hunk context missing | `hunk @@ ... @@ failed to apply: context not found` |
| all | unknown operation | `unknown operation: "X"` |
| all | OS read/write failure | error passed through as-is |

## Testing

New file: `internal/tools/file_edit_test.go`

Coverage:
- Happy path for each of the three operations
- `replace_text` with `old_string` not found
- `replace_lines` with out-of-range line numbers
- `replace_lines` with empty `content` (deletion)
- `apply_patch` with a malformed or non-applicable patch
- Unknown operation name

## Decisions

- **Flat schema over nested args**: matches the style of existing tools (`file_read`, `file_write`) and is more reliably generated by LLMs than nested objects.
- **First-occurrence-only for replace_text**: predictable and safe; avoids mass replacement surprises.
- **Pure Go patch parser**: preserves the project's CGO_ENABLED=0 / static binary constraint.
- **No `exec diff/patch`**: keeps the binary self-contained and avoids platform dependency on system utilities.
